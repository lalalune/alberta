================================================================================
THE ALBERTA PLAN: 12-STEP QUICK REFERENCE
================================================================================

STEP 1: Representation I — Continual Supervised Learning (Fixed Features)
─────────────────────────────────────────────────────────────────────────
Problem: Linear supervised learning y = w·x + b with non-stationary targets
Key Innovation: Per-feature adaptive step-sizes (IDBD, Autostep) + online normalization
Success When: Tuning-free methods beat hand-tuned LMS on sparse-relevance & drift tasks
Referenced: Sutton 1992 (IDBD), Mahmood et al. 2012 (Autostep)
Status in Framework: COMPLETE

STEP 2: Representation II — Supervised Feature Finding
─────────────────────────────────────────────────────────────────────────
Problem: Learn to create new features (products, compositional) under fixed budget
Key Innovation: Smart generation + candidate testing + feature ranking + deletion
Success When: Feature discovery beats fair MLPs on out-of-hypothesis-class tasks
Referenced: Sutton et al. 2011 (GVF/Horde foundation)
Status in Framework: PARTIAL (UPGD strong on synthetic; struggles external benchmarks)

STEP 3: Prediction I — Continual GVF Prediction Learning
─────────────────────────────────────────────────────────────────────────
Problem: Predict multi-step GVF returns with learned features on sequential data
Key Innovation: Eligibility traces + per-demon γ/λ + TD learning + off-policy
Success When: GVF prediction + feature discovery works on sequential tasks
Referenced: Sutton et al. 2011 (Horde), Sutton 1988 (TD learning)
Status in Framework: PARTIAL (GVF prediction works; off-policy convergence needs work)

STEP 4: Control I — Continual Actor-Critic Control
─────────────────────────────────────────────────────────────────────────
Problem: Learn both policy (actor) and value function (critic) in k-arm → contextual → sequential
Key Innovation: Critic from Steps 1-3 + actor policy learning in parallel
Success When: Actor-critic learns robustly without divergence across settings
Referenced: Sutton & Barto (2018)
Status in Framework: PARTIAL (SARSA on-policy; off-policy actor-critic not yet)

STEP 5: Prediction II — Average-Reward GVF Learning
─────────────────────────────────────────────────────────────────────────
Problem: Extend GVF prediction to continuing (undiscounted) objectives
Key Innovation: Learn average reward ρ + differential value V^ρ separately
Success When: Average-reward algorithms work with minimal changes from Step 3
Referenced: Classical average-reward DP (Puterman & Brumelle)
Status in Framework: MINIMAL (skeleton exists; not integrated with Steps 1-3)

STEP 6: Control II — Continuing Control Benchmarks
─────────────────────────────────────────────────────────────────────────
Problem: Create/adapt non-episodic benchmark environments for testing
Key Innovation: River Swim, Access-Control, Jellybean World, GARNET (continuing versions)
Success When: Algorithms trained on benchmarks converge to stationary policies
Referenced: Bhatnagar et al. (GARNET), classical continuing domains
Status in Framework: MINIMAL (benchmarks exist but not central to framework)

STEP 7: Planning I — Average-Reward Planning (Asynchronous DP)
─────────────────────────────────────────────────────────────────────────
Problem: Use world model for offline value improvement in continuing setting
Key Innovation: Asynchronous DP (update states in any order) + function approximation
Success When: Planning converges to ε-optimal policies; DP order affects speed
Referenced: Bertsekas (async DP), Sutton et al. 2008 (Dyna + linear FA)
Status in Framework: PRIMITIVE (planning interface exists; not full async DP)

STEP 8: Prototype-AI I — One-Step Model-Based RL
─────────────────────────────────────────────────────────────────────────
Problem: Integrate Steps 1-7 into first complete agent (no temporal abstraction yet)
Key Innovation: Foreground learning loop + background model + feature-guided planning
Components: (a) state construction, (b) one-step model, (c) feature finding w/ model feedback,
            (d) feature ranking, (e) cycle linking model↔features→planning, (f) search control
Success When: Integrated system runs; planning improves policy over model-free
Referenced: Sutton (1990) Dyna; Steps 1-7 foundations
Status in Framework: PRIMITIVE (scaffolding present; not integrated end-to-end)

STEP 9: Planning II — Search Control & Exploration
─────────────────────────────────────────────────────────────────────────
Problem: Decide which states to prioritize in asynchronous DP with function approx.
Key Innovation: Prioritized sweeping + small backups + uncertainty-aware ordering (novel w/ FA)
Success When: Good state ordering improves convergence; uncertainty correlates with value
Referenced: Peng & Williams 1993 (prioritized sweep), Van Seijen & Sutton 2013 (small backups)
Status in Framework: PRIMITIVE (planning interface; no sophisticated search control)

STEP 10: Prototype-AI II — The STOMP Progression (Temporal Abstraction)
─────────────────────────────────────────────────────────────────────────
Problem: Introduce options (temporally abstract actions) for long-horizon planning
Innovation: Features→SubTasks→Options→Models→Planning cycle (STOMP)
           Each high-utility feature becomes a subtask (goal), which becomes an option
Components: Feature ranking → Subtask definition → Option learning (off-policy) → Model learning
Success When: Options reduce effective horizon; option models learned accurately
Referenced: Sutton, Precup, Singh 1999 (options); Sutton et al. 2022 (STOMP)
Status in Framework: PRIMITIVE (STOMP framework sketched; off-policy learning incomplete)

STEP 11: Prototype-AI III — Oak Architecture (Lifelong Abstraction)
─────────────────────────────────────────────────────────────────────────
Problem: Continually assess & replace all abstractions (features, subtasks, options, models)
Innovation: Utility assessment loop; deletion of low-value elements; promotion of candidates
            Option Keyboard: index options by chord vectors; combine options
Components: (a) utility tracking for all elements, (b) reranking cycle, (c) deletion/promotion,
            (d) option chord combinations
Success When: Lifelong learning; weak options auto-deleted; hierarchy improves over time
Referenced: Barreto et al. 2019 (option keyboard); Sutton et al. 2022
Status in Framework: PRIMITIVE (lifecycle interfaces sketched; not continuous)

STEP 12: Prototype-IA — Intelligence Amplification (Dual-Agent System)
─────────────────────────────────────────────────────────────────────────
Problem: One agent augments another (human or AI) through prediction & policy sharing
Innovation: Two versions:
            (a) Exo-cerebellum: Prototype-IA provides GVF predictions (perceptual augmentation)
            (b) Exo-cortex: Provides action recommendations & policies (decision augmentation)
            Both learn what info/actions the partner values
Components: Dual-agent learning, communication bandwidth mgmt, online adaptation to partner
Success When: Partner's speed/accuracy improves with IA; works human-agent & agent-agent
Referenced: Pilarski et al. 2022 (GVF coordination); Ashby/Licklider/Engelbart (IA history)
Status in Framework: PRIMITIVE (GVF prediction exists; no dual-agent coordination)

KEY METRICS: TEMPORAL UNIFORMITY & FOREGROUND/BACKGROUND

FOREGROUND (Every Time Step — Must Complete Before Acting)
──────────────────────────────────────────────────────────
1. Perception: Construct state from observation
2. Reactive Policy: Select action
3. Value Functions: Evaluate state (learning signal)
4. Learning: Update weights/features/traces

BACKGROUND (Asynchronous — Runs Continuously in Parallel)
──────────────────────────────────────────────────────────
1. World Model Learning: Improve transition estimates
2. Planning: Asynchronous DP value improvement
3. Feature Construction: Generate & test candidates
4. Abstraction Management: Update utilities, delete low-value elements

TEMPORAL UNIFORMITY: All algorithms operate on every time step (or continuously).
No special "training phase"; no offline batches; no episodic resets (unless task-required).

================================================================================
ARCHITECTURAL DIFFERENCES: ALBERTA PLAN vs. STANDARD RL

Aspect              | Standard RL              | Alberta Plan
───────────────────┼─────────────────────────┼──────────────────────────────
Horizon             | Episodic (resets)       | Continuing (indefinite)
Feature Repr.       | Fixed/pre-trained       | Continually constructed & replaced
Planning            | Optional, offline       | Continuous, asynchronous, integrated
Abstraction         | Manual (hierarchical RL)| Learned from feature ranking
Meta-learning       | Outer loop              | Per-step (step-sizes, feature selection)
Deployment          | Sim-to-real + tuning    | Daemon-ready (real-time single-step loop)
Reward Signal       | Single global           | Average-reward + GVF cumulants

================================================================================
EXPECTED SYSTEM AT COMPLETION (STEPS 1-12)

State/Observations
        ↓
   FOREGROUND (Real-Time Control Loop)
   ├─ Perception (recurrent state update)
   ├─ Feature Extraction (learned, ranked by utility)
   ├─ Reactive Policy (greedy over value function)
   └─ Value Functions (multiple, per-demon GVF)
        ↓ (action)
        
   ENVIRONMENT (Reward + Next Observation)
   
   BACKGROUND (Asynchronous, Parallel)
   ├─ Model Learning (one-step transition)
   ├─ Planning (async DP over options)
   ├─ Feature Management (utility assessment → promotion/deletion)
   ├─ STOMP Cycle (Features → SubTasks → Options → Models)
   └─ Oak Lifecycle (continuous refinement of all abstractions)

Result: A single integrated agent that learns and acts continually,
discovers its own hierarchies, and can be augmented with IA partners.

================================================================================
ROBOT/REAL-WORLD DEPLOYMENT REQUIREMENTS

Time Budget per Loop: <100 ms (10 Hz typical robotics cycle)
  ├─ Perception: <10 ms (state construction)
  ├─ Policy: <5 ms (action selection)
  ├─ Learning: <10 ms (weight updates, feature construction)
  └─ Buffer: <75 ms (planning, model updates — can be interrupted)

Daemon Compatibility:
  ├─ Single-step API: predict(obs) → value, update(error, features)
  ├─ Incremental: No batches, no replay, no special initialization
  ├─ Graceful degradation: Works with or without planning/features

Continuing Operation:
  ├─ No episodic resets (or learns to recognize natural resets)
  ├─ Handles non-stationarity (feature reranking, model adaptation)
  ├─ Memory bounded (feature budget enforced; old options deleted)
